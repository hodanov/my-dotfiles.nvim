# Write Blog Entry

## Goal

入力素材をもとに、ユーザーと対話しながらブログ記事を完成させる。

## Workflow

### Phase 1: 入力の把握

1. 引数にファイルパスがある場合はその内容を読み込む
2. 引数にテーマの説明がある場合はそれを入力として使う
3. 引数が無い場合はセッション内の直近の壁打ち内容を入力として使う
4. 入力が不十分な場合は推測で埋めず、ユーザーに確認する

確認すべきポイント:

- 記事の対象読者は誰か
- 記事で伝えたいことは何か
- 技術的な背景や具体的なエピソードはあるか

### Phase 2: アウトライン提案（承認ゲート）

入力を分析し、以下をユーザーに提案する:

- **タイトル候補**（1〜3案）
- **要約**（1〜3行）
- **アウトライン**（見出し構成）

```markdown
## 提案

### タイトル候補
1. <案1>
2. <案2>

### 要約
<1〜3行の要約>

### アウトライン
1. はじめに
2. <見出し2>
3. <見出し3>
4. ...
5. まとめ
```

**このフェーズでユーザーの承認を得るまで Phase 3 に進まない。**
フィードバックがあれば修正して再提案する。

### Phase 3: ファイル解決

1. `draft_entries/` 配下のファイルが引数で指定されている → そのファイルを更新対象とする
2. Phase 1 の入力ファイルが `draft_entries/` 配下にある → そのファイルを更新対象とする
3. いずれでもない → ユーザーにファイルパスを確認する

既存ファイルを更新する場合は Front Matter の `Title` のみ更新し、
`EditURL`, `URL`, `Date` などの自動生成フィールドは変更しない。

### Phase 4: 執筆

1. 承認されたアウトラインに沿って記事を書く
2. 記事フォーマットは [template.md](template.md) に従う

### Phase 5: 校正（サブエージェント委譲）

執筆完了後、校正を `textlint-fixer` サブエージェントに委譲する。

1. Agent tool で `textlint-fixer` を起動する
   - プロンプト:

     ```txt
     以下のファイルを校正してください。
     ファイルパス: <対象ファイルの絶対パス>
     ```

2. `textlint-fixer` から Textlint Report を受け取る
3. Report をユーザーに表示する
4. ステータスが FAIL の場合、未解決エラーをユーザーに提示し対応を相談する

## 執筆ルール

- `AGENTS.md` の textlint ルール・禁止表現に従う
- 1文は130文字以内、読点は1文中4つまで
- 技術用語は正式名称を使う（`GCP` → `Google Cloud` など）
- 改行は半角スペース2つ、段落区切りは空行
- サムネイルと `[:contents]` は記事冒頭の指定位置に配置する
- 具体的なコード、コマンド、設定値を入れて実用的な記事にする
- 入力素材に無い事実を捏造しない

## Gotchas

- `entries/` は公開済み記事なので絶対に変更しない。執筆は `draft_entries/` で行う
- Front Matter の `Title` 以外（`EditURL`, `URL`, `Date`）は自動生成されるため手動で変更しない
- はてなブログ記法 `[:contents]` は Markdown の見出しから目次を自動生成する。削除しない
- `[f:id:DMMTech:...]` 形式の画像参照は textlint の `skipPatterns` でスキップされる
- 画像は `images/YYYY/MM/` に配置し、記事からは相対パスで参照する
